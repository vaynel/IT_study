# 보안 아키텍처 다이어그램

## 개념

### 정의
보안 아키텍처 다이어그램은 시스템의 보안 구조를 표현하는 다이어그램입니다. 보안 구역, 방화벽, 보안 정책, 접근 제어 구조를 시각화하여 보안 아키텍처를 설계하고 문서화합니다.

### 핵심 특징
- **보안 중심**: 보안 구조에 초점
- **보안 구역**: DMZ, 내부 네트워크 등 보안 구역 표현
- **방화벽**: 방화벽 배치 및 규칙
- **접근 제어**: 접근 제어 구조
- **암호화**: 데이터 암호화 구조

### 주요 구성 요소

#### 보안 구역
- **정의**: 보안 레벨별 구역
- **표기**: 영역 박스 또는 색상
- **예시**: DMZ, 내부 네트워크, 관리 네트워크

#### 방화벽
- **정의**: 네트워크 보안 장비
- **표기**: 방패 모양 또는 박스
- **예시**: 웹 방화벽, 네트워크 방화벽

#### 보안 정책
- **정의**: 보안 규칙 및 정책
- **표기**: 라벨 또는 주석
- **예시**: 접근 정책, 암호화 정책

#### 접근 제어
- **정의**: 사용자 및 시스템 접근 제어
- **표기**: 화살표 또는 라벨
- **예시**: 인증, 인가, 권한 관리

### 1. 보안 요구사항이 높은 시스템

- **예시**:

- **과한 이유**:
  - 보안 구조가 단순하여 문서화 불필요

### 2. 완전 관리형 서비스
- **상황**: 클라우드 완전 관리형 서비스만 사용

- **과한 이유**:
  - 보안이 클라우드 제공자에 의해 관리됨

# 위협 모델링 다이어그램

## 개념

### 정의
위협 모델링 다이어그램은 시스템에 대한 위협을 식별하고 분석하는 다이어그램입니다. 공격 벡터, 위협 액터, 취약점, 대응 방안을 시각화하여 보안 위협을 관리합니다.

### 핵심 특징
- **위협 중심**: 보안 위협에 초점
- **공격 벡터**: 공격 경로 시각화
- **위협 액터**: 공격자 유형
- **취약점**: 시스템 취약점
- **대응 방안**: 위협 대응 전략

### 주요 구성 요소

#### 위협 액터
- **정의**: 공격을 수행하는 주체
- **표기**: 사람 아이콘 또는 박스
- **예시**: 외부 해커, 내부자, 경쟁사

#### 공격 벡터
- **정의**: 공격 경로
- **표기**: 화살표
- **예시**: 네트워크 공격, 애플리케이션 공격, 물리적 공격

#### 취약점
- **정의**: 시스템 취약점
- **표기**: 경고 아이콘 또는 박스
- **예시**: 취약한 인증, SQL 인젝션, XSS

#### 자산
- **정의**: 보호해야 할 자산
- **표기**: 박스
- **예시**: 데이터베이스, 사용자 정보, 비즈니스 로직

#### 대응 방안
- **정의**: 위협 대응 전략
- **표기**: 방패 아이콘 또는 박스
- **예시**: 방화벽, 암호화, 접근 제어

# 테스트 다이어그램

## 개념

### 정의
테스트 다이어그램은 테스트 계획과 테스트 케이스를 시각화하는 다이어그램입니다. 테스트 시나리오, 테스트 커버리지, 테스트 흐름을 표현하여 테스트를 계획하고 실행합니다.

### 핵심 특징
- **테스트 중심**: 테스트 계획과 실행에 초점
- **테스트 케이스**: 테스트 케이스 시각화
- **커버리지**: 테스트 커버리지 표현
- **테스트 흐름**: 테스트 실행 흐름
- **결과 분석**: 테스트 결과 분석

### 주요 구성 요소

#### 테스트 케이스
- **정의**: 개별 테스트 케이스
- **표기**: 박스
- **예시**: 단위 테스트, 통합 테스트, 시스템 테스트

#### 테스트 시나리오
- **정의**: 테스트 실행 시나리오
- **표기**: 플로우차트 또는 시퀀스 다이어그램
- **예시**: 사용자 시나리오, 비즈니스 시나리오

#### 테스트 커버리지
- **정의**: 테스트 커버리지 범위
- **표기**: 맵 또는 그래프
- **예시**: 코드 커버리지, 기능 커버리지

#### 테스트 결과
- **정의**: 테스트 실행 결과
- **표기**: 상태 아이콘 또는 그래프
- **예시**: 통과, 실패, 블로킹

# 서비스 블루프린트

## 개념

### 정의
서비스 블루프린트(Service Blueprint)는 서비스를 제공하는 전체 프로세스를 시각화하는 다이어그램입니다. 고객 행동, 전면 행동, 백스테이지 행동, 지원 프로세스를 표현하여 서비스를 설계하고 개선합니다.

### 핵심 특징
- **서비스 중심**: 서비스 제공 프로세스에 초점
- **다층 구조**: 고객, 전면, 백스테이지, 지원 프로세스
- **터치포인트**: 고객과의 접촉점
- **서비스 개선**: 서비스 개선 기회 발견

### 주요 구성 요소

#### 고객 행동
- **정의**: 고객의 행동
- **표기**: 상단 레인
- **예시**: 서비스 요청, 정보 확인, 피드백

#### 전면 행동
- **정의**: 고객과 직접 접촉하는 행동
- **표기**: 두 번째 레인
- **예시**: 고객 응대, 서비스 제공

#### 백스테이지 행동
- **정의**: 고객과 직접 접촉하지 않는 행동
- **표기**: 세 번째 레인
- **예시**: 데이터 처리, 내부 프로세스

#### 지원 프로세스
- **정의**: 서비스를 지원하는 프로세스
- **표기**: 하단 레인
- **예시**: 시스템, 인프라, 외부 서비스

# 이벤트 스토밍

## 개념

### 정의
이벤트 스토밍(Event Storming)은 도메인 이벤트를 중심으로 비즈니스 프로세스를 모델링하는 워크샵 기법입니다. 이벤트, 명령, 액터, 애그리게이트를 시각화하여 도메인 모델을 설계합니다.

### 핵심 특징
- **이벤트 중심**: 도메인 이벤트 중심 모델링
- **협업**: 다양한 이해관계자 협업
- **빠른 모델링**: 빠른 도메인 모델링
- **DDD**: 도메인 주도 설계(DDD) 지원

### 주요 구성 요소

#### 도메인 이벤트
- **정의**: 비즈니스에서 발생하는 이벤트
- **표기**: 주황색 스티커
- **예시**: 주문 생성됨, 결제 완료됨, 배송 시작됨

#### 명령
- **정의**: 이벤트를 발생시키는 명령
- **표기**: 파란색 스티커
- **예시**: 주문 생성, 결제 처리, 배송 시작

#### 액터
- **정의**: 명령을 수행하는 주체
- **표기**: 노란색 스티커
- **예시**: 사용자, 관리자, 시스템

#### 애그리게이트
- **정의**: 도메인 모델의 경계
- **표기**: 큰 박스
- **예시**: 주문, 상품, 고객

---

## 사용하는 곳

### 1. 도메인 모델링
- 도메인 모델 설계
- 비즈니스 프로세스 이해
- DDD 설계

### 2. 요구사항 분석
- 비즈니스 요구사항 분석
- 도메인 전문가와 협업

# 도메인 모델 다이어그램

## 개념

### 정의
도메인 모델 다이어그램은 비즈니스 도메인의 개념과 관계를 표현하는 다이어그램입니다. 엔티티, 값 객체, 애그리게이트, 도메인 서비스를 시각화하여 도메인 모델을 설계합니다.

### 핵심 특징
- **도메인 중심**: 비즈니스 도메인에 초점
- **DDD**: 도메인 주도 설계 지원
- **비즈니스 언어**: 비즈니스 언어 사용
- **도메인 로직**: 도메인 로직 표현

### 주요 구성 요소

#### 엔티티
- **정의**: 고유 식별자를 가진 도메인 객체
- **표기**: 박스
- **예시**: 주문, 고객, 상품

#### 값 객체
- **정의**: 값으로 식별되는 객체
- **표기**: 박스 (값 객체 표시)
- **예시**: 금액, 주소, 날짜

#### 애그리게이트
- **정의**: 일관성 경계를 가진 엔티티 그룹
- **표기**: 큰 박스
- **예시**: 주문 애그리게이트, 상품 카탈로그

#### 도메인 서비스
- **정의**: 엔티티에 속하지 않는 도메인 로직
- **표기**: 박스 (서비스 표시)
- **예시**: 가격 계산 서비스, 할인 계산 서비스
