## C4 모델 다이어그램

### 개념
C4 모델은 소프트웨어 아키텍처를 4가지 관점(Context, Container, Component, Code)에서 표현하는 모델입니다. 각 레벨은 다른 관심사를 가진 이해관계자에게 적합한 추상화 수준을 제공합니다.

### 주요 레벨
1. **시스템 컨텍스트 다이어그램 (Level 1)**: 시스템과 외부 액터의 관계
2. **컨테이너 다이어그램 (Level 2)**: 애플리케이션의 컨테이너 구조
3. **컴포넌트 다이어그램 (Level 3)**: 컨테이너 내부의 컴포넌트 구조
4. **코드 다이어그램 (Level 4)**: 클래스나 함수 레벨의 상세 구조

### 사용하는 곳
- **아키텍처 문서화**: 소프트웨어 아키텍처를 계층적으로 문서화
- **이해관계자 소통**: 다양한 관심사를 가진 이해관계자와 소통
- **아키텍처 설계**: 시스템 아키텍처 설계 및 검토
- **온보딩**: 신규 개발자에게 시스템 구조 설명
- **아키텍처 리뷰**: 아키텍처 결정 검토

### 어디에 사용하면 유리한지
✅ **대규모 시스템**: 복잡한 시스템의 구조를 단계적으로 이해
✅ **다양한 이해관계자**: 기술자와 비기술자 모두와 소통
✅ **마이크로서비스**: 여러 서비스로 구성된 시스템
✅ **레거시 시스템 문서화**: 기존 시스템의 구조 파악
✅ **아키텍처 의사결정**: 아키텍처 결정을 문서화
✅ **프로젝트 초기 단계**: 시스템 구조를 설계할 때
✅ **리팩토링 계획**: 구조 개선 계획 수립

### 어떤 경우에는 과한 경우인지
❌ **간단한 프로젝트**: 단일 애플리케이션으로 구성된 작은 프로젝트
❌ **프로토타입**: 빠르게 변경되는 초기 프로토타입
❌ **일회성 스크립트**: 임시 스크립트나 유틸리티
❌ **단순한 정적 웹사이트**: 복잡한 구조가 없는 경우
❌ **개인 프로젝트**: 개인만 사용하는 작은 프로젝트

### 개발 문서에 포함될 경우
- **시스템 아키텍처 문서**: 전체 시스템 아키텍처
- **설계 문서**: 시스템 설계 및 구조
- **온보딩 문서**: 신규 개발자 교육 자료
- **아키텍처 결정 기록 (ADR)**: 아키텍처 결정 문서
- **기술 문서**: 기술 스택 및 구조 설명
- **프로젝트 제안서**: 시스템 구조 제안

## ER 다이어그램

### 개념
ER 다이어그램(Entity-Relationship Diagram)은 데이터베이스의 엔티티(개체), 속성, 그리고 엔티티 간의 관계를 시각적으로 표현하는 다이어그램입니다. 관계형 데이터베이스 설계의 핵심 도구입니다.

### 주요 구성 요소
- **엔티티 (Entity)**: 데이터베이스의 테이블에 해당하는 개념적 객체 (사각형)
- **속성 (Attribute)**: 엔티티의 특성 (타원형)
  - 기본키(Primary Key): 밑줄
  - 외래키(Foreign Key): 점선 밑줄
- **관계 (Relationship)**: 엔티티 간의 연관 (마름모)
- **카디널리티**: 
  - 1:1 (일대일)
  - 1:N (일대다)
  - N:M (다대다)
- **약한 엔티티**: 다른 엔티티에 의존적인 엔티티 (이중 사각형)

### 사용하는 곳
- **데이터베이스 설계**: 관계형 데이터베이스 스키마 설계
- **데이터 모델링**: 비즈니스 도메인의 데이터 구조 모델링
- **데이터베이스 리팩토링**: 기존 스키마 개선 계획
- **데이터베이스 문서화**: 기존 데이터베이스 구조 문서화
- **요구사항 분석**: 데이터 요구사항 분석
- **데이터베이스 마이그레이션**: 데이터베이스 이전 계획

### 어디에 사용하면 유리한지
✅ **관계형 데이터베이스**: RDBMS를 사용하는 모든 프로젝트
✅ **복잡한 데이터 구조**: 많은 테이블과 복잡한 관계
✅ **정규화가 중요한 경우**: 데이터 정규화 설계
✅ **데이터 무결성**: 참조 무결성 설계가 중요한 경우
✅ **대규모 데이터베이스**: 많은 엔티티와 관계가 있는 경우
✅ **데이터베이스 리팩토링**: 기존 구조 개선
✅ **팀 협업**: 데이터베이스 설계를 팀과 공유

### 어떤 경우에는 과한 경우인지
❌ **NoSQL 데이터베이스**: 문서형이나 키-값 저장소
❌ **단순한 데이터 구조**: 테이블이 1~2개인 경우
❌ **프로토타입 단계**: 빠르게 변경되는 초기 단계
❌ **임시 데이터베이스**: 일회성 데이터 저장
❌ **단순한 파일 저장**: 파일 기반 저장소

### 개발 문서에 포함될 경우
- **데이터베이스 설계서**: 데이터베이스 스키마 설계
- **데이터 모델 문서**: 도메인 데이터 모델
- **시스템 설계서**: 데이터 구조 설계
- **데이터베이스 문서**: 기존 데이터베이스 구조
- **마이그레이션 계획서**: 데이터베이스 이전 계획
- **API 문서**: 데이터 구조 설명 (특히 GraphQL)

## 네트워크 다이어그램

### 개념
네트워크 다이어그램은 네트워크의 토폴로지, 장비, 연결, 그리고 데이터 흐름을 시각적으로 표현하는 다이어그램입니다. 물리적 네트워크 구조와 논리적 네트워크 구조를 모두 표현할 수 있습니다.

### 주요 구성 요소
- **네트워크 장비**: 
  - 라우터: 원형
  - 스위치: 사각형
  - 방화벽: 방패 모양
  - 로드 밸런서: 다이아몬드
- **서버**: 박스
- **클라이언트**: PC 아이콘
- **네트워크 링크**: 연결선
- **네트워크 세그먼트**: 클라우드 모양
- **IP 주소/서브넷**: 라벨

### 사용하는 곳
- **네트워크 설계**: 새로운 네트워크 인프라 설계
- **네트워크 문서화**: 기존 네트워크 구조 문서화
- **트러블슈팅**: 네트워크 문제 진단
- **보안 설계**: 네트워크 보안 구조 설계
- **용량 계획**: 네트워크 용량 및 대역폭 계획
- **재해 복구 계획**: 네트워크 백업 및 복구 구조
- **네트워크 마이그레이션**: 네트워크 구조 변경 계획

### 어디에 사용하면 유리한지
✅ **복잡한 네트워크**: 여러 서브넷과 장비가 있는 경우
✅ **분산 시스템**: 여러 데이터센터나 클라우드 리전
✅ **하이브리드 클라우드**: 온프레미스와 클라우드 혼합
✅ **보안 요구사항이 높은 경우**: DMZ, 방화벽 규칙 등
✅ **네트워크 통합**: 여러 네트워크를 통합하는 경우
✅ **네트워크 최적화**: 네트워크 성능 개선 계획
✅ **규정 준수**: 네트워크 구조를 문서화해야 하는 경우

### 어떤 경우에는 과한 경우인지
❌ **단순한 로컬 네트워크**: 홈 네트워크나 작은 오피스
❌ **단일 서버**: 하나의 서버만 있는 경우
❌ **클라우드 관리형 서비스**: 네트워크가 완전히 추상화된 경우
❌ **개발 환경**: 개발자 로컬 환경
❌ **프로토타입**: 임시 네트워크 구성

### 개발 문서에 포함될 경우
- **인프라 설계서**: 네트워크 인프라 설계
- **네트워크 문서**: 네트워크 구조 및 구성
- **보안 설계서**: 네트워크 보안 구조
- **운영 매뉴얼**: 네트워크 운영 가이드
- **재해 복구 계획서**: 네트워크 복구 절차
- **변경 관리 문서**: 네트워크 변경 계획

## 인프라스트럭처 다이어그램

### 개념
인프라스트럭처 다이어그램은 IT 인프라의 물리적 및 논리적 구조를 표현하는 다이어그램입니다. 서버, 스토리지, 네트워크, 보안 장비 등 전체 인프라 구조를 시각화합니다.

### 주요 구성 요소
- **서버**: 물리적/가상 서버 (박스)
- **스토리지**: SAN, NAS, 클라우드 스토리지 (원통)
- **네트워크 장비**: 라우터, 스위치, 방화벽
- **로드 밸런서**: 트래픽 분산 장비
- **데이터베이스**: DB 서버 (원통)
- **백업 시스템**: 백업 스토리지 및 시스템
- **모니터링 시스템**: 모니터링 도구
- **연결**: 네트워크 연결 및 의존성

### 사용하는 곳
- **인프라 설계**: 새로운 인프라 설계
- **인프라 문서화**: 기존 인프라 구조 문서화
- **용량 계획**: 리소스 용량 계획
- **하드웨어 요구사항**: 필요한 하드웨어 스펙 정의
- **재해 복구 계획**: 백업 및 복구 인프라
- **보안 설계**: 보안 인프라 구조
- **클라우드 마이그레이션**: 클라우드 이전 계획

### 어디에 사용하면 유리한지
✅ **대규모 인프라**: 여러 서버와 장비가 있는 경우
✅ **하이브리드 인프라**: 온프레미스와 클라우드 혼합
✅ **고가용성 시스템**: 장애 대비 인프라
✅ **규정 준수**: 인프라 구조를 문서화해야 하는 경우
✅ **인프라 최적화**: 리소스 최적화 계획
✅ **비용 계획**: 인프라 비용 계획
✅ **보안 감사**: 보안 구조 검토

### 어떤 경우에는 과한 경우인지
❌ **단순한 클라우드 서비스**: 완전 관리형 서비스만 사용
❌ **개발 환경**: 개발자 로컬 환경
❌ **소규모 프로젝트**: 서버가 1~2개인 경우
❌ **프로토타입**: 임시 인프라
❌ **서버리스 아키텍처**: 인프라가 완전히 추상화된 경우

### 개발 문서에 포함될 경우
- **인프라 설계서**: 전체 인프라 구조
- **하드웨어 요구사항 문서**: 필요한 하드웨어 스펙
- **운영 매뉴얼**: 인프라 운영 가이드
- **재해 복구 계획서**: 인프라 복구 절차
- **보안 설계서**: 보안 인프라 구조
- **비용 계획서**: 인프라 비용 분석

## 데이터 플로우 다이어그램

### 개념
데이터 플로우 다이어그램은 시스템 내에서 데이터가 어떻게 흐르고 변환되는지를 표현하는 다이어그램입니다. 데이터의 소스, 처리 과정, 저장소, 목적지를 시각화합니다.

### 주요 구성 요소
- **프로세스**: 데이터를 처리하는 기능 (원형)
- **데이터 저장소**: 데이터를 저장하는 곳 (열린 사각형)
- **외부 엔티티**: 시스템 외부의 데이터 소스/목적지 (사각형)
- **데이터 흐름**: 데이터의 이동 경로 (화살표)
- **데이터 딕셔너리**: 데이터 구조 정의

### 사용하는 곳
- **시스템 분석**: 데이터 흐름 분석
- **ETL 프로세스**: 데이터 추출, 변환, 로딩 프로세스
- **데이터 파이프라인**: 데이터 처리 파이프라인 설계
- **시스템 통합**: 여러 시스템 간 데이터 흐름
- **데이터 웨어하우스**: 데이터 웨어하우스 설계
- **비즈니스 프로세스**: 데이터 중심 비즈니스 프로세스

### 어디에 사용하면 유리한지
✅ **데이터 중심 시스템**: 데이터 처리와 변환이 핵심인 시스템
✅ **ETL 프로세스**: 복잡한 데이터 변환 프로세스

✅ **데이터 웨어하우스**: 데이터 수집 및 분석 시스템
✅ **비즈니스 인텔리전스**: 데이터 분석 프로세스

## DFD (Data Flow Diagram)

### 개념
DFD는 시스템의 데이터 흐름을 여러 레벨로 세분화하여 표현하는 다이어그램입니다. 컨텍스트 다이어그램에서 시작하여 점진적으로 상세화합니다.

### 주요 레벨
- **Level 0 (컨텍스트 다이어그램)**: 시스템과 외부 엔티티의 관계
- **Level 1**: 주요 프로세스로 분해
- **Level 2+**: 각 프로세스를 더 상세히 분해

### 주요 구성 요소
- **프로세스**: 데이터 처리 (원형, 번호 포함)
- **데이터 저장소**: 데이터 저장 (열린 사각형, D로 시작)
- **외부 엔티티**: 외부 시스템/사용자 (사각형)
- **데이터 흐름**: 데이터 이동 (화살표, 라벨 포함)

### 사용하는 곳
- **시스템 분석**: 구조적 시스템 분석
- **요구사항 분석**: 데이터 요구사항 분석
- **시스템 설계**: 데이터 중심 시스템 설계
- **시스템 문서화**: 기존 시스템의 데이터 흐름 문서화
- **시스템 리팩토링**: 데이터 흐름 개선 계획

### 어디에 사용하면 유리한지
✅ **구조적 분석 방법론**: 전통적인 구조적 분석
✅ **복잡한 데이터 처리**: 여러 단계의 데이터 처리
✅ **레거시 시스템 분석**: 기존 시스템의 데이터 흐름 파악
✅ **시스템 통합**: 여러 시스템의 데이터 흐름 통합
✅ **데이터 중심 설계**: 데이터 흐름이 핵심인 시스템

### 어떤 경우에는 과한 경우인지
❌ **객체지향 설계**: OOP 중심의 설계 방법론
❌ **애자일 개발**: 빠른 반복 개발
❌ **간단한 시스템**: 데이터 흐름이 단순한 경우
❌ **현대적인 마이크로서비스**: 서비스 간 단순 통신

### 개발 문서에 포함될 경우
- **시스템 분석 문서**: 구조적 시스템 분석
- **요구사항 명세서**: 데이터 요구사항
- **시스템 설계서**: 데이터 중심 설계
- **레거시 시스템 문서**: 기존 시스템 분석

### 주요 구성 요소

- **입력 (Input)**: 왼쪽에서 들어오는 데이터
- **출력 (Output)**: 오른쪽으로 나가는 데이터
- **제어 (Control)**: 위에서 들어오는 제어 신호
- **메커니즘 (Mechanism)**: 아래에서 제공되는 리소스/도구

### 사용하는 곳
- **시스템 분석**: 시스템 기능 분석
- **요구사항 분석**: 기능 요구사항 정의
- **인터페이스 설계**: 시스템 인터페이스 설계
- **시스템 문서화**: 시스템 기능 문서화

### 어디에 사용하면 유리한지
✅ **제어 시스템**: 제어 로직이 중요한 시스템
✅ **자동화 시스템**: 자동화 프로세스
✅ **하드웨어 인터페이스**: 하드웨어와 소프트웨어 인터페이스
✅ **시스템 분석**: 시스템 기능을 체계적으로 분석

### 어떤 경우에는 과한 경우인지
❌ **단순한 웹 애플리케이션**: 제어와 메커니즘이 명확하지 않은 경우
❌ **일반적인 비즈니스 애플리케이션**: 4가지 관점이 불필요한 경우

### 개발 문서에 포함될 경우
- **시스템 분석 문서**: 시스템 기능 분석
- **요구사항 명세서**: 기능 요구사항
- **인터페이스 설계서**: 시스템 인터페이스

- **IDEF3**: 프로세스 모델링 (Process Modeling)

### 사용하는 곳

- **데이터 모델링**: 데이터 구조 모델링
- **엔터프라이즈 아키텍처**: 기업 전체 아키텍처

### 어디에 사용하면 유리한지
✅ **구조적 분석 방법론**: 전통적인 구조적 분석
✅ **대규모 시스템**: 복잡한 대규모 시스템
✅ **정부/군사 프로젝트**: IDEF가 요구되는 프로젝트
✅ **엔터프라이즈 아키텍처**: 기업 전체 아키텍처 모델링

### 어떤 경우에는 과한 경우인지
❌ **애자일 개발**: 빠른 반복 개발
❌ **소규모 프로젝트**: 간단한 프로젝트

### 개발 문서에 포함될 경우
