# 시스템 다이어그램

## 개념

### 정의
시스템 다이어그램은 시스템의 전체 구조와 구성 요소를 표현하는 다이어그램입니다. 시스템의 경계, 서브시스템, 인터페이스를 시각화하여 시스템을 이해하고 설계합니다.

### 핵심 특징
- **시스템 중심**: 시스템 전체 구조 표현
- **경계 정의**: 시스템의 범위와 경계 명시
- **서브시스템**: 시스템을 서브시스템으로 분해
- **인터페이스**: 서브시스템 간 인터페이스 표현
- **계층 구조**: 계층적 시스템 구조 표현

### 주요 구성 요소

#### 시스템
- **정의**: 전체 시스템
- **표기**: 외부 박스
- **예시**: 전체 애플리케이션, 전체 시스템

#### 서브시스템
- **정의**: 시스템의 주요 구성 요소
- **표기**: 내부 박스
- **예시**: 인증 서브시스템, 데이터 처리 서브시스템

#### 인터페이스
- **정의**: 서브시스템 간 통신 경로
- **표기**: 연결선
- **예시**: API, 메시지 버스, 이벤트

#### 경계
- **정의**: 시스템의 범위
- **표기**: 시스템 박스 경계
- **예시**: 시스템 경계, 외부 시스템과의 경계

#### 외부 시스템
- **정의**: 시스템과 상호작용하는 외부 시스템
- **표기**: 시스템 외부 박스
- **예시**: 외부 API, 서드파티 서비스

- **유리한 점**:

## 어떤 경우에는 과한 경우인지

### 1. 단일 애플리케이션
- **상황**: 하나의 애플리케이션만 있는 경우
- **예시**: 
  - 단순한 웹 애플리케이션
  - 개인 프로젝트
- **과한 이유**:
  - 구조가 단순하여 문서화 불필요
  - 코드로 충분히 이해 가능

### 2. 간단한 시스템
- **상황**: 구조가 단순한 시스템
- **예시**: 
  - 간단한 스크립트
  - 단순한 유틸리티
- **과한 이유**:
  - 서브시스템 개념이 없음
  - 문서화 불필요

### 3. 프로토타입
- **상황**: 빠르게 변경되는 초기 프로토타입
- **예시**: 
  - MVP 개발
  - 개념 검증
- **과한 이유**:
  - 구조가 자주 변경됨
  - 문서 업데이트 비용 높음

# 논리 다이어그램

## 개념

### 정의
논리 다이어그램은 시스템의 논리적 구조를 표현하는 다이어그램입니다. 물리적 구현과 독립적으로 시스템의 논리적 구성 요소와 관계를 표현합니다.

### 핵심 특징
- **논리적 관점**: 물리적 구현과 독립적
- **추상화**: 구현 세부사항 제거
- **개념적 설계**: 개념적 시스템 구조
- **플랫폼 독립**: 플랫폼에 독립적
- **아키텍처 중심**: 아키텍처 관점에서 표현

### 주요 구성 요소

#### 논리 컴포넌트
- **정의**: 기능적 구성 요소
- **표기**: 박스
- **예시**: 비즈니스 로직, 데이터 접근 계층, 프레젠테이션 계층

#### 논리 관계
- **정의**: 컴포넌트 간 논리적 관계
- **표기**: 연결선
- **예시**: 의존성, 호출 관계, 데이터 흐름

#### 데이터 흐름
- **정의**: 논리적 데이터 흐름
- **표기**: 화살표
- **예시**: 데이터 전송, 데이터 변환

#### 계층
- **정의**: 논리적 계층 구조
- **표기**: 수직 배치
- **예시**: 3계층 아키텍처, N계층 아키텍처

#### 인터페이스
- **정의**: 논리적 인터페이스
- **표기**: 원 또는 릴레이
- **예시**: API 인터페이스, 서비스 인터페이스

# 물리 다이어그램

## 개념

### 정의
물리 다이어그램은 시스템의 물리적 배치와 인프라를 표현하는 다이어그램입니다. 실제 서버, 네트워크, 스토리지 등 물리적 구성 요소를 표현합니다.

### 핵심 특징
- **물리적 관점**: 실제 하드웨어와 인프라 표현
- **배치 정보**: 물리적 배치 위치 정보
- **하드웨어**: 실제 하드웨어 장비 표현
- **인프라**: 네트워크, 스토리지 등 인프라 표현
- **운영 중심**: 운영 환경 중심 표현

### 주요 구성 요소

#### 서버
- **정의**: 물리적/가상 서버
- **표기**: 박스
- **예시**: 웹 서버, 애플리케이션 서버, 데이터베이스 서버

#### 네트워크
- **정의**: 물리적 네트워크 연결
- **표기**: 연결선
- **예시**: LAN, WAN, 인터넷 연결

#### 스토리지
- **정의**: 물리적 스토리지
- **표기**: 원통
- **예시**: SAN, NAS, 로컬 스토리지

#### 위치
- **정의**: 물리적 배치 위치
- **표기**: 그룹핑 또는 라벨
- **예시**: 데이터센터, 클라우드 리전, 오피스

#### 하드웨어
- **정의**: 실제 하드웨어 장비
- **표기**: 장비 아이콘
- **예시**: 라우터, 스위치, 방화벽, 로드 밸런서

아키텍처 다이어그램은 시스템 아키텍처를 다양한 관점에서 표현하는 다이어그램입니다. 시스템의 구조, 구성 요소, 관계를 시각화하여 아키텍처를 이해하고 설계합니다.

### 핵심 특징
- **다양한 관점**: 여러 관점에서 아키텍처 표현
- **구조 중심**: 시스템 구조에 초점
- **패턴 적용**: 아키텍처 패턴 시각화
- **의사결정 기록**: 아키텍처 의사결정 기록
- **문서화**: 아키텍처 문서화

- **목적**: 논리적 구조 표현

- **용도**: 개념적 설계, 아키텍처 설계

#### 물리적 아키텍처

- **포함**: 배포 단위, 배치 구조
- **용도**: 배포 계획, 운영 계획

#### 통합 아키텍처

- **포함**: 시스템 간 통합, 인터페이스

# 시스템 컨텍스트 다이어그램

## 개념

### 정의
시스템 컨텍스트 다이어그램은 시스템과 외부 액터의 관계를 표현하는 다이어그램입니다. 시스템의 경계, 외부 의존성, 액터와의 관계를 시각화합니다.

### 핵심 특징
- **시스템 중심**: 시스템을 중심으로 표현
- **경계 정의**: 시스템 경계 명확히 정의
- **외부 의존성**: 외부 시스템과의 의존성 표현
- **액터 관계**: 사용자 및 액터와의 관계

---

## 사용하는 곳

### 1. 시스템 경계 정의
- 시스템 범위 정의
- 시스템 경계 명확화
- 외부 의존성 식별

### 2. 아키텍처 개요
- 시스템 아키텍처 개요
- 이해관계자 소통
- 시스템 이해도 향상

### 3. 요구사항 분석
- 시스템 요구사항 분석
- 외부 요구사항 식별
- 시스템 범위 결정

---

## 어디에 사용하면 유리한지

### 1. 시스템 설계 초기
- **상황**: 시스템 설계 초기 단계
- **유리한 점**:
  - 시스템 경계 정의
  - 외부 의존성 식별
  - 이해관계자 소통

### 2. 복잡한 시스템
- **상황**: 많은 외부 의존성이 있는 시스템
- **유리한 점**:
  - 외부 의존성 시각화
  - 시스템 경계 명확화

---

## 어떤 경우에는 과한 경우인지

### 1. 단순한 시스템
- **상황**: 외부 의존성이 없는 단순한 시스템
- **과한 이유**:
  - 구조가 단순하여 문서화 불필요

---

  - 외부 시스템
  
- **시스템 상세**

## 도구

### 시스템 다이어그램 도구
- **Draw.io**: 다양한 시스템 다이어그램 지원
- **Lucidchart**: 시스템 다이어그램 템플릿
- **Visio**: 기업 표준 도구
- **Structurizr**: C4 모델 기반 시스템 다이어그램

### 아키텍처 도구
- **ArchiMate**: 엔터프라이즈 아키텍처 모델링
- **C4 Model**: 소프트웨어 아키텍처 문서화
- **PlantUML**: 텍스트 기반 아키텍처 다이어그램

## 모범 사례

### 시스템 다이어그램
1. 시스템 경계 명확히 정의
2. 서브시스템 적절히 분해
3. 인터페이스 명확히 표현
4. 계층 구조 유지

### 논리/물리 다이어그램
1. 논리와 물리 구분 명확히
2. 적절한 추상화 수준 유지
3. 일관된 표기법 사용
4. 정기적 업데이트

---

## 결론

시스템 분석 및 설계 다이어그램은 시스템을 다양한 관점에서 분석하고 설계하는 강력한 도구입니다. 시스템 다이어그램은 전체 구조를, 논리 다이어그램은 논리적 구조를, 물리 다이어그램은 물리적 구조를 각각 표현합니다.

각 다이어그램은 고유한 목적과 사용 사례를 가지고 있으며, 시스템의 특성과 분석/설계 단계에 맞는 적절한 다이어그램을 선택하는 것이 중요합니다. 적절한 상황에서 사용하면 시스템 이해도와 설계 품질을 크게 향상시킬 수 있습니다.
